# Engineering Review

这份文档不是为“找借口”，而是为说明当前系统已经完成什么，以及距离更稳健的工程实现还差什么。

## Strengths

### 1. Problem decomposition is correct

项目没有停留在单进程 demo，而是拆成了三个职责不同的端：

- `Sensor` 负责输入
- `Car` 负责控制与展示
- `DataSrv` 负责汇聚与转发

这说明设计重点放在了系统边界，而不是单个按钮功能。

### 2. Communication paths are layered

项目同时处理了两类通信：

- 串口通信：传感器到车辆
- TCP/IP 通信：车辆到服务端、车辆到车辆

这让系统天然具备“设备侧 + 网络侧”的综合性。

### 3. The project closes the loop

不是只做到“收到数据”为止，而是把这些环节串起来了：

- 采集
- 上传
- 服务端展示
- 预警/反馈
- 日志存档
- 条件查询
- 车辆互联

这对课程项目和展示型仓库都很重要。

## Current Engineering Debt

### 1. Sensor data is still simulated

三类传感器目前主要通过随机数生成数据，适合验证链路，但不适合说明真实设备接入已经完成。

这不是缺陷本身，关键是要明确它是为了缩小第一阶段问题规模而做的取舍。

### 2. Local paths are hard-coded

代码里保留了若干课程时期的桌面图片和 CSV 固定路径，例如：

- 桌面图片资源路径
- 本机 CSV 文件落盘路径

这会影响他人在别的机器上直接复现，也说明当时更偏向功能打通而不是部署抽象。

### 3. Protocol design is still ad-hoc

当前通信大量依赖字符串和端口号，没有统一消息头、类型字段、长度字段和版本机制。

课程阶段这样做可以快速推进，但如果继续扩展：

- 消息兼容会变差
- 调试成本会升高
- 服务端行为会更难维护

### 4. UI and business logic are tightly coupled

MFC 对话框文件同时承担了：

- 控件初始化
- 串口管理
- socket 连接
- 数据展示
- 文件读写
- 部分业务流程控制

这种写法能快速做出结果，但后续重构和测试都会变难。

### 5. Threading is functional but rough

项目已经开始尝试使用 `CreateThread` 和 `std::thread` 处理接收与中转，这说明作者意识到了异步问题。

但当前线程模型仍然偏粗放：

- 共享状态边界不清
- 生命周期管理不完整
- 线程安全机制不足

这类问题在课程阶段常见，但也是最值得继续提升的地方。

## Why These Gaps Are Still Valuable

这个仓库适合展示的，不只是“做出来了”，还包括：

- 能把问题先缩到可解范围
- 知道哪些实现是为了先验证闭环
- 能指出当前方案为什么还能继续工程化

换句话说，这个项目体现的是：

`先把系统跑起来，再识别真正的工程瓶颈。`

## If I Continue This Project

我会优先做四件事：

1. 把配置抽离：IP、端口、文件路径全部从代码挪到配置层
2. 把协议结构化：定义消息类型、载荷字段、错误码和版本号
3. 把逻辑解耦：UI、串口、网络、存储分层
4. 把模拟数据替换为真实设备接入或可重放数据源

这样改完以后，这个项目就会从“课程答辩系统”更接近“可维护的工程样例”。
