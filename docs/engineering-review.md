# Engineering Review

这份文档里，我想说清楚两件事：

- 这个系统已经做到什么
- 它距离更稳健的工程实现还差什么

我不想把这份文档写成“替项目辩护”，也不想写成纯夸奖总结。它更像是我对自己这次实现的一次复盘。

## 1. What I Think I Got Right

### 1.1 I solved a system problem, not a single-point feature

我没有停留在单进程 demo，而是把系统拆成了三个职责不同的端：

- `Sensor` 负责输入
- `Car` 负责控制和展示
- `DataSrv` 负责汇聚和转发

这说明我当时的重点放在了系统边界，而不是只补某个局部功能。

### 1.2 I handled both device-side and network-side communication

这个项目同时处理了两类通信：

- 串口通信：传感器到车辆
- TCP/IP 通信：车辆到服务端、车辆到车辆

这让整个项目天然不只是一个界面程序，而是一个跨设备和跨进程的协同系统。

### 1.3 I built a complete loop

我没有在“收到数据”这一步停下，而是把这些环节串起来了：

- 采集
- 上传
- 服务端展示
- 反馈
- 日志存档
- 条件查询
- 车辆互联

对课程项目来说，这比单点功能实现更有说明力。

## 2. What Is Still Rough

### 2.1 Sensor data is still simulated

现在三类传感器数据主要还是通过随机数生成。

这适合先验证链路，但不代表真实设备接入已经完成。这个取舍本身没有问题，前提是我必须把它说清楚：我当时优先解决的是“链路先跑通”，而不是“硬件先接完”。

### 2.2 Local paths are still hard-coded

代码里保留了一些课程时期的本机图片路径和 CSV 路径。

这意味着：

- 换一台机器直接运行会遇到环境问题
- 当时实现重点偏向功能打通，而不是部署抽象

这类问题不影响展示思路，但会影响复现体验。

### 2.3 Protocol design is still ad-hoc

当前很多消息仍然依赖字符串和端口号，没有统一的消息头、类型字段、长度字段和版本机制。

这在课程阶段能快速推进，但如果继续扩展，会带来这些问题：

- 兼容性会变差
- 调试成本会升高
- 服务端逻辑会更难维护

### 2.4 UI and business logic are tightly coupled

MFC 对话框文件里同时承担了很多职责：

- 控件初始化
- 串口管理
- socket 连接
- 数据展示
- 文件读写
- 部分业务流程控制

这能很快把结果做出来，但后续重构和测试都会变难。

### 2.5 Threading works, but it is still rough

我已经开始用 `CreateThread` 和 `std::thread` 处理接收和中转，这说明我当时意识到了异步问题。

但现在的线程模型还比较粗：

- 共享状态边界不够清楚
- 生命周期管理不够完整
- 线程安全机制不足

这正是后续最值得继续补的地方之一。

## 3. Why These Gaps Still Matter Positively

我觉得这个仓库有价值，不只是因为“它能跑”，还因为它能真实展示这类项目通常会经历的路径：

- 先把问题缩到可解范围
- 先做出能工作的系统
- 再识别哪些地方只是课程阶段解法
- 再决定后续工程化优先级

所以这些不足不是我要藏起来的部分，反而是我希望读者能看到的部分，因为它们说明我知道系统目前处在什么阶段。

## 4. If I Continue This Project

如果我继续做，我会优先做这四件事：

1. 把配置抽离：IP、端口、文件路径都从代码挪到配置层
2. 把协议结构化：定义消息类型、载荷字段、错误码和版本号
3. 把逻辑解耦：把 UI、串口、网络、存储拆分
4. 把模拟数据替换为真实设备接入或可重放数据源

做完这几步以后，这个项目就会从“课程答辩系统”更接近“可维护的工程样例”。
