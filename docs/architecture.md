# Architecture Notes

这份文档是我对整个系统拆分方式的说明。重点不是“模块名是什么”，而是我为什么这样拆，以及每一端分别解决什么问题。

## 1. Why I Split It This Way

我没有把这个课设做成一个单体程序，而是拆成了三个端：

- `Sensor`
- `Car`
- `DataSrv`

这样拆的原因很直接：

- 传感器采集、车辆交互、服务端汇聚本来就是三类不同职责
- 我希望每一端都能单独验证，而不是所有逻辑都堆在一个工程里
- 这样后面无论接真实硬件、换协议，还是重构线程模型，都更容易继续推进

## 2. System Split

### Sensor

在这个端里，我主要解决的是“数据从哪里来”的问题。

这里负责：

- 模拟速度、车距、行人数量三类感知数据
- 通过串口控件输出数据
- 给车辆端提供统一输入来源

我在代码里做的对应拆分是：

- `CSensor` 提供统一抽象接口
- `CSpeedSensor`、`CDistanceSensor`、`CPerSensor` 分别生成不同类型的数据
- `02_TCPClientDlg.cpp` 负责串口初始化、启动停止和定时发送控制

### Car

在这个端里，我主要解决的是“怎么把采集、展示、上传、查询和交互串起来”的问题。

这里负责：

- 接收传感器侧数据
- 展示速度、车距、行人等状态
- 作为 TCP 客户端与中心服务器通信
- 提供日志导出、CSV 查询、车辆间消息通信界面

我在代码里把这些功能主要放在：

- `02_TCPClientDlg.cpp`
  主交互窗口和总控入口
- `DataSearch.cpp`
  CSV 查询与条件过滤
- `CarCommu.cpp`
  基于辅助服务器的车辆间通信

### DataSrv

在这个端里，我主要解决的是“怎么做接入、回传和中转”的问题。

这里负责：

- 接收车辆端连接
- 记录客户端 IP、端口和在线状态
- 回发服务端消息
- 通过辅助端口做车辆配对和中继

我在代码里把职责分成：

- `ServerSocket.cpp`
  主服务器的 Accept、连接登记和广播
- `MyServerSocket.cpp`
  辅助服务器的端口配对和消息中转
- `02_TCPServerDlg.cpp`
  服务端控制界面

## 3. Main Data Flow

我设计的主数据流是这样的：

1. `Sensor` 端初始化串口和传感器类型。
2. 传感器数据通过串口进入 `Car` 端。
3. `Car` 端通过 TCP 连接 `DataSrv` 主服务器 `:8888`。
4. `DataSrv` 接收连接并记录客户端状态。
5. `Car` 和 `DataSrv` 之间进行消息收发，形成主链路闭环。
6. 车辆端可以把数据落到 CSV，再通过查询窗口筛选和回看。

这条主链路对我来说最重要，因为它证明这个系统不只是“本地能显示数据”，而是已经形成了从采集到上传再到反馈和留存的完整路径。

## 4. Vehicle-to-Vehicle Communication

车车通信这部分，我没有直接做点对点固定地址连接，而是用了一个更适合课程阶段验证的办法：

- 把车辆端口号当作简化寻址手段
- 通过辅助服务器 `:8811` 做配对和中转

流程是：

1. 车辆连接辅助服务器 `:8811`
2. 服务器记录当前连接端口
3. 一辆车提交目标端口
4. 服务器在在线列表里找到对应客户端
5. 完成配对后转发后续消息

我这样设计的原因是：

- 主上传链路和车车通信本来就是两类问题
- 我想先验证机制，而不是一开始就追求复杂协议
- 服务端集中维护状态，比客户端各自管理地址簿更适合这类课程项目

## 5. Why This Split Matters

如果把所有逻辑都写进一个程序里，系统当然也能演示，但会很难说明“每一部分到底在解决什么问题”。

拆成 `Sensor / Car / DataSrv` 以后，我得到的好处是：

- 每一端职责边界更清楚
- 串口、TCP、日志查询和消息中转都能分别验证
- 更容易向老师说明系统结构，而不是只展示界面
- 后续接真实硬件或升级协议时，改动范围更可控

## 6. If I Continue This Project

如果我继续把这个项目往更正式的方向推进，我会按这个顺序做：

1. 把硬编码 IP、端口和 CSV 路径改成配置文件
2. 把字符串消息升级为结构化协议
3. 把串口、网络、UI、存储逻辑拆成独立模块
4. 引入线程安全的数据通道，而不是直接共享窗口状态
5. 把 CSV 留档升级为可查询的数据模型或轻量数据库

也就是说，这个项目现在最有价值的地方，不只是它已经跑起来了，而是它的下一步工程化路径已经很清楚。
