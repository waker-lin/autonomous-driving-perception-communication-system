# Architecture Notes

## 1. System Split

### Sensor

职责：

- 模拟速度、距离、行人数量三类感知数据
- 通过串口控件输出数据
- 为车辆端提供统一输入来源

实现线索：

- `CSensor` 提供统一抽象接口
- `CSpeedSensor`、`CDistanceSensor`、`CPerSensor` 分别负责不同数据生成逻辑
- `02_TCPClientDlg.cpp` 中承担串口初始化、启动停止和定时发送控制

### Car

职责：

- 接收传感器侧数据
- 展示速度、车距、行人等状态
- 作为 TCP 客户端与中心服务器通信
- 提供日志导出、CSV 查询、车辆间消息通信界面

实现线索：

- `02_TCPClientDlg.cpp` 是主交互窗口和总控入口
- `DataSearch.cpp` 负责 CSV 查询与条件过滤
- `CarCommu.cpp` 负责基于辅助服务器的车辆间通信

### DataSrv

职责：

- 接收车辆端连接
- 记录客户端 IP/端口和在线状态
- 将服务端消息回发给客户端
- 通过辅助端口做车辆配对和中继

实现线索：

- `ServerSocket.cpp` 负责主服务器的 Accept、连接登记和广播
- `MyServerSocket.cpp` 负责辅助服务器的端口配对和中转
- `02_TCPServerDlg.cpp` 提供启动、停止、发送和列表展示界面

## 2. Main Data Flow

1. `Sensor` 端初始化串口与传感器类型。
2. 传感器数据通过串口进入 `Car` 端界面。
3. `Car` 端通过 TCP 连接 `DataSrv` 主服务器 `:8888`。
4. `DataSrv` 接收连接、记录客户端、展示在线状态。
5. `Car` 与 `DataSrv` 之间进行消息收发，形成演示闭环。
6. 车辆端可将数据落到 CSV，并通过查询窗口做筛选。

## 3. Vehicle-to-Vehicle Communication

这个功能的设计重点是：不直接维护复杂地址簿，而是把车辆端口号作为简化寻址手段，通过辅助服务器做配对和消息转发。

流程：

1. 车辆连接辅助服务器 `:8811`
2. 辅助服务器记录连接端口
3. 一辆车提交目标端口
4. 服务器在已连接列表中查找匹配客户端
5. 建立配对后转发后续消息

这个实现虽然简单，但展示了一个重要工程思路：

- 主业务链路与实验性交互链路分离
- 服务端承担状态集中管理与中转职责
- 在课程阶段先用最小协议验证机制，再考虑正式协议设计

## 4. Why The Split Matters

如果把所有逻辑都塞进单个程序，系统会很快变成“能演示但难说明”的状态。拆成 `Sensor / Car / DataSrv` 三端以后，优势是：

- 每一端职责边界更清楚
- 可以分别验证串口、TCP、日志查询和中转机制
- 更容易向老师说明系统架构，而不是只展示界面
- 更适合后续替换真实硬件或升级协议

## 5. Suggested Evolution Path

如果继续做成更像正式项目，建议按这个顺序推进：

1. 把硬编码 IP、端口和 CSV 路径改成配置文件
2. 把字符串消息升级为结构化协议
3. 把串口、网络、UI、存储逻辑拆成独立模块
4. 引入线程安全的数据通道，而不是直接共享窗口状态
5. 把 CSV 留档升级为可查询的数据模型或轻量数据库
